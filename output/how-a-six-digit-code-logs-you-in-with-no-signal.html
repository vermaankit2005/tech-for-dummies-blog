How a Six-Digit Code Logs You In With No Signal

Your phone's in airplane mode. The website's a thousand miles away. You type six digits and it lets you in — and the two of them never exchange a word. So how do they agree?

The site wants six digits.

You know the drill. You typed your password, and now there's a second box: enter the code from your authenticator app. So you swipe over to the app, and there it is — six digits, with a little ring draining down beside them. 281 942. You type them in. You're back in your account.

Now glance at the top of your phone. Airplane mode. No signal, no Wi-Fi, nothing. That app didn't fetch the code from anywhere — it had nowhere to fetch it from. And the website you're logging into, running on a server you couldn't point to on a map, somehow expected those exact six digits. Two machines that never spoke just agreed on a number.

They have spoken, actually. Once. And it was a while ago.

Rewind to the day you set this up. The site showed you a QR code — that grid of black squares — and told you to scan it with the app. You assumed it was a link. It wasn't.

It was a secret. Buried in that QR code was a string of random characters — something like JBSWY3DPEHPK3PXP, written in a format called Base32 so it survives being passed around without garbling. The site generated it, showed it to you once, and your app swallowed it. Now both of them hold the same secret: the site keeps its copy, your phone keeps its copy. And from that moment on they go quiet — neither one ever sends the secret to the other again. The technical name for that little provisioning packet inside the QR is an otpauth URI; once it's scanned, its job is done.

That shared secret is half the trick. Here's the other half.

They don't stay in sync by talking. They stay in sync by both watching the same clock.

Your phone knows what time it is. So does the server. Not the time you read off the lock screen — Unix time, the plain count of seconds that have ticked by since the first instant of 1970. Right now that number is past 1.7 billion, and it climbs by one every second, everywhere on Earth, in lockstep.

Both sides take that number and chop it into blocks of 30 seconds — they divide by 30 and throw away the remainder. So for any given half-minute, your phone and the server are holding the exact same block number, without checking with each other once. The thing they have in common isn't a connection. It's the time.

Now they each do the same piece of arithmetic. Your phone takes the secret and the current block number and feeds both into a function called HMAC-SHA1 — a one-way scrambler. You can pour two things in and get a fixed jumble of bytes out, but you can't run it backwards: hand someone the jumble and they can't recover the secret that made it. Out comes 20 bytes of noise.

Then your phone trims it down. It reads the very last sliver of that noise to pick a starting point, snips out four bytes from there, reads them as one number, and keeps the last six digits. That step has a name — dynamic truncation — and the result is your code: 281 942.

The server, a thousand miles away, holds the same secret and the same block number. It runs the same scrambler, makes the same cut, keeps the same six digits. It got 281 942 too. Not because your phone told it. Because they did identical math on identical inputs, and there was only ever one answer.

Identical inputs, identical math, one possible answer — computed twice, a thousand miles apart, with nothing passing between them.

That's why it works in airplane mode. Your phone was never going to ask anyone. It already held everything it needed — the secret from setup, and the time, which it carries with it everywhere.

The ring draining beside the code is that 30-second block running out. When it empties, the block number ticks up by one, both sides feed the new number through the scrambler, and a fresh six digits surfaces on each end at the same instant. A code that's expired isn't rejected because it got "used up" — it's rejected because the block it belonged to is gone, and the math now lands on something else.

Which raises a problem. What if your phone's clock and the server's clock don't quite agree? Off by a few seconds, and you'd be computing block 58 142 001 while the server has already moved to 58 142 002 — different number in, different six digits out, login denied. So the server tilts the odds in your favor. When it checks your code, it doesn't only compute the current block; it also computes the one just before and the one just after, and accepts a match on any of the three. That ±1 cushion is why a code you typed a hair late still slides through — and why each one really lives for up to about 90 seconds, not 30.

The server checks the block before and the block after, so one code survives up to ~90 seconds of clock drift between your phone and the server.

There's one thing the code doesn't know — and attackers live in that gap. Those six digits prove that whoever typed them holds the secret and can read a clock. They prove nothing about where they're being typed. The code is a bearer token: it works for anyone who's holding it, at the right moment, on any site that asks for it.

⚠️ This is the one way it falls. If a convincing fake login page tricks you into entering your code, the person running that page can relay it to the real site within the 90-second window and walk straight in as you. The code did its job perfectly — it matched the clock. It just never knew which door it was opening. This is exactly the gap that newer logins — the FIDO2 keys and passkeys built into your phone — close, by binding the login to the real website's name, something a typed-in number can't do.

You finish typing. The site lets you in. 281 942 blinks out, and a new code slides up to take its place.

Your phone has been in airplane mode the entire time. It never reached for a signal, never phoned the server, never needed to. It just knew a secret it has never once said out loud — and it knew what time it is.
